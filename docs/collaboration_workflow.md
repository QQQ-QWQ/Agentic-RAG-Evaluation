# 小组分工与协作流程

本文档合并原 `team_roles.md` 与 `collaboration_workflow.md` 的内容，统一说明三名成员的职责分工、近期交付物、日常同步方式、每周验收机制和冲突处理规则。后续小组协作以本文档为准。

## 1. 总体分工

| 角色                 | 成员   | 主要方向                                                | 核心交付物                                              |
| -------------------- | ------ | ------------------------------------------------------- | ------------------------------------------------------- |
| A 方法与统筹负责人   | 赵启行 | 研究主线、C0-C4 设计、Prompt、消融实验、报告和 PPT      | 实验配置说明、Prompt、消融分析、报告主线                |
| B 数据与评测负责人   | 唐宁   | 知识库、测试集、参考答案、Benchmark、人工评分、失败案例 | `questions.csv`、`references.csv`、评分表、失败案例 |
| C 工程与 Demo 负责人 | 李金航 | 环境、代码、检索、工具、日志、Demo、C5 横向参考         | 可运行代码、日志、结果表、Demo                          |

分工可以根据进度调整，但调整后必须更新本文档，避免出现“大家都以为别人会做”的空档。

## 2. 成员 A：赵启行

赵启行为组长，主要负责方法设计、整体推进和最终表达。组长不一定做最多代码，但要保证项目不散、不虚、不失控。

主要职责：

- 把握项目定位：本项目不是单纯产品型学习助手，而是 Agentic RAG 技术增强与评测研究。
- 设计 C0-C4 对比配置，明确每个配置只新增什么能力。
- 设计 query rewrite、task planning、self-check 等核心 Prompt。
- 设计核心消融实验，如 no-rewrite、no-self-check、no-tool。
- 统筹结题报告和答辩 PPT 主线。
- 每周检查项目是否偏离研究主线。

近期交付物：

- [ ] 明确 C0-C4 配置差异。
- [ ] 写出实验假设和五类核心指标。
- [ ] 编写 query rewrite 与 self-check Prompt 初稿。
- [ ] 设计至少三个核心消融实验。
- [ ] 检查 C1、C2 是否按统一接口输出日志。

协作边界：

- 和唐宁对齐测试题是否能验证研究问题。
- 和李金航对齐配置是否能被代码实现。
- 不直接替代其他成员长期完成其负责模块，但要帮助拆解任务和验收产出。

## 3. 成员 B：唐宁

唐宁主要负责数据、测试集、评测和失败案例分析。这个角色决定项目最终能不能从“能跑 Demo”变成“能量化证明有效”。

主要职责：

- 收集和整理知识库材料。
- 建立 `documents.csv`。
- 编写自建测试集。
- 标注参考答案、证据 chunk 和所需工具。
- 筛选 10-15 条公共 Benchmark 参考子集。
- 进行人工评分和失败案例整理。

近期交付物：

- [ ] 收集第一批知识库材料。
- [ ] 编写首批 20 条测试任务。
- [ ] 为每题标注参考答案和证据。
- [ ] 建立评分规则草稿。
- [ ] 为 C1 准备一批模糊、口语化、非规范查询样例。

协作边界：

- 不根据系统输出随意修改测试集。
- 正式实验前冻结测试集。
- 与赵启行一起确认题目是否服务研究问题。
- 与李金航一起确认数据格式是否能被程序读取。

## 4. 成员 C：李金航

李金航主要负责工程实现、运行日志、实验脚本和 Demo。这个角色决定项目能不能稳定运行、复现和展示。

主要职责：

- 维护 Python 和 uv 环境。
- 实现文档解析、chunk、检索、rerank。
- 实现四类工具：文件读取、代码执行、计算器、表格分析。
- 实现 C3/C4 Agentic RAG 流程。
- 保存运行日志和结果表。
- 准备轻量 Demo。
- 完成 Dify 或 RAGFlow 横向参考。

近期交付物：

- [ ] 确认 uv 环境可运行。
- [ ] 巩固 C0 文档解析和检索代码。
- [ ] 配合 C1 输出统一日志。
- [ ] 准备 C2 的 BM25、hybrid retrieval 和 rerank。
- [ ] 设计运行日志字段和结果保存方式。

协作边界：

- 不把 API Key 写进代码。
- 不写死个人电脑路径。
- 新增依赖必须使用 `uv add`，并同步更新 `pyproject.toml` 和 `uv.lock`。
- 代码输出必须能保存到 `runs/`。

## 5. 组长每日检查重点

赵启行每天重点看五件事：

1. 当前任务是否围绕 C0-C4 主线。
2. 每个人是否有明确交付物。
3. 数据、代码、日志、结果是否放在正确目录。
4. 结论是否能被测试集和指标验证。
5. 是否出现范围过大、进度失控或单人压力过大的情况。

组长要避免只问“做了吗”，更应该问：

- 产出文件在哪里？
- 怎么运行？
- 怎么验收？
- 失败了怎么记录？
- 对报告结论有什么帮助？

## 6. 每日同步机制

每天建议同步 10-15 分钟，不开长会。每个人按以下模板汇报：

```text
日期：
成员：
昨天完成：
产出文件：
今天计划：
当前卡点：
需要协助：
```

组长只做三件事：

1. 判断任务是否偏离主线。
2. 判断产出是否可检查。
3. 决定是否需要调整分工。

不要使用“差不多”“快好了”这类状态。每个任务必须能看到产出文件、运行命令、结果表或文档记录。

## 7. 每周验收机制

每周至少做一次 60 分钟复盘。复盘不只是聊天，必须检查具体产出。

每周检查项：

| 检查项 | 说明                                       |
| ------ | ------------------------------------------ |
| 环境   | 新成员或另一台电脑能否 `uv sync` 后运行  |
| 数据   | 知识库和测试集是否增加，参考答案是否可判定 |
| 代码   | 是否有可运行脚本，是否能复现结果           |
| 日志   | 是否保存运行过程，字段是否统一             |
| 结果   | 是否有结果表、对比表或案例                 |
| 报告   | 本周工作是否能写进报告                     |

每周复盘模板：

```text
本周目标：
本周完成：
可检查产出：
未完成内容：
未完成原因：
是否影响总进度：
下周最重要任务：
需要删减的内容：
需要老师或学长帮助的问题：
是否需要调整分工：
```

组长每周至少更新一次 `docs/experiment_notes.md`，让后续报告有材料可写。

## 8. 阶段验收标准

| 阶段   | 验收标准                                       |
| ------ | ---------------------------------------------- |
| 环境   | 三个人都能运行 `uv run python --version`     |
| 数据集 | 至少 20 条首批测试题，每题有参考答案和证据     |
| C0     | 能检索、回答、输出引用和日志                   |
| C1     | 能记录原问题、改写问题、检索查询，并比较 C0/C1 |
| C2     | 能输出混合检索和 rerank 结果                   |
| 工具   | 四类工具都能单独运行并保存日志                 |
| C3     | 能展示任务规划、多轮检索和 self-check          |
| C4     | 能展示工具选择、工具调用和工具结果验证         |
| 评测   | 有 C0-C4 主结果表和至少 3 个消融结果           |
| Demo   | 能展示完整 Agentic RAG 技术链路                |
| 报告   | 能说明问题、方法、实验、结果、边界             |

## 9. 任务状态规范

所有任务建议使用以下状态：

```text
未开始 / 进行中 / 待验收 / 已完成 / 有风险
```

任务进入“已完成”前，必须满足：

- 有对应文件或运行结果。
- 组长或另一名成员能复现或检查。
- 关键问题记录在 `docs/experiment_notes.md` 或相关文档中。

## 10. 冲突处理

### 10.1 代码冲突

处理原则：

1. 不要直接删除别人的代码。
2. 先看冲突文件内容。
3. 找对应负责人确认。
4. 合并后必须运行相关脚本或测试。

### 10.2 任务冲突

如果两个人同时改同一类文件，优先按模块拆开：

| 文件类型           | 推荐负责人       |
| ------------------ | ---------------- |
| `data/testset/*` | B                |
| `src/*`          | C                |
| `prompts/*`      | A                |
| `configs/*`      | A/C              |
| `docs/*`         | A 主导，全员补充 |

### 10.3 进度冲突

如果时间不够，删减顺序：

1. C4-Adaptive 选做内容。
2. 实时网页搜索。
3. 完整 Dify + RAGFlow 双系统对比，只保留一个。
4. 复杂图表。

不能删：

1. C0-C4 主线。
2. 测试集。
3. 核心消融。
4. 日志。
5. 失败案例。

## 11. 协作底线

- 不用微信传代码，GitHub 仓库是唯一同步来源。
- 不提交 `.env`、API Key、`.venv/`、临时日志和大体积索引文件。
- 不在没有同步 main 的情况下长期独立开发。
- 不为了让结果好看而修改测试集或删除失败案例。
- 不把 Demo 能跑等同于实验完成，必须有日志、指标和失败分析。
